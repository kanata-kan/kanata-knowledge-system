## Layer-Based vs Feature-Based

### Layer-Based (Traditional)

```
src/
├── controllers/
├── services/
├── models/
├── routes/
└── utils/
```

Good for small projects. Breaks down at scale because related code is scattered.

### Feature-Based (Recommended for Scale)

```
src/
├── features/
│   ├── auth/
│   │   ├── auth.controller.ts
│   │   ├── auth.service.ts
│   │   ├── auth.routes.ts
│   │   └── auth.types.ts
│   ├── users/
│   │   ├── users.controller.ts
│   │   ├── users.service.ts
│   │   └── users.types.ts
├── shared/
│   ├── middleware/
│   ├── utils/
│   └── types/
└── config/
```

## Key Principles

1. **Colocation** — Keep related code together
2. **Bounded modules** — Each feature is self-contained
3. **Shared last** — Only extract to shared when needed by 3+ features
4. **Flat hierarchy** — Max 3 levels of nesting
5. **Index exports** — Each module has a public API via index.ts

## Frontend (Next.js) Pattern

```
src/
├── app/           # Routes only — thin page files
├── components/
│   ├── ui/       # Generic reusable (Button, Badge, Card)
│   └── [feature]/ # Feature-specific components
├── lib/           # Utilities, data access
├── types/         # Shared types
└── content/       # Static content (MDX, JSON)
```

## The "Screaming Architecture" Test

Your folder structure should scream what the app does, not what framework it uses. Someone looking at your `src/` should understand the domain.

Bad: `controllers/`, `services/`, `repositories/`
Good: `auth/`, `payments/`, `notifications/`, `knowledge/`
